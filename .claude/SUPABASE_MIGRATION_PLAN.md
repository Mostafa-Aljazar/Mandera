# PocketBase → Supabase Full Migration

## Context

The app (`apps/web`) currently talks directly to a self-hosted PocketBase instance (`apps/pocketbase`) from client components — there is no server layer at all today (`src/app/api` and `src/actions` both exist but are empty; no TanStack Query anywhere). The user wants to fully replace PocketBase with Supabase, remove `apps/pocketbase` entirely, and in doing so introduce a proper server layer: Server Actions in `src/actions` that call Supabase directly (no intermediate API routes for internal calls), consumed from client components via TanStack Query for caching/loading/error state.

This plan covers **every PocketBase consumer in the codebase** — all 19 tables/collections, all 3 auth realms, every page/component/context/hook that calls `pb.collection(...)`, the one realtime subscription, the file-upload flow, and the 10 PocketBase JS hooks encoding business logic that has no home yet. Execution proceeds module by module in the order below so each slice can be verified independently, but nothing is deferred to a later *planning* pass — the full roadmap is fixed now.

### Key decisions (confirmed with user)
- **Actions call Supabase directly** — no internal API-route hop. `src/app/api` stays empty unless a genuinely external caller (webhook, mobile app) shows up later.
- **Auth**: single `auth.users` (Supabase Auth) + a `public.profiles` table carrying `role` (`master_admin | company_super_admin | company_employee`), `company_id`, `employee_id`, `name`. Replaces PocketBase's 3 separate auth collections (`master_admins`, `company_super_admins`, `company_employees`). RLS policies key off `auth.uid()` joined to `profiles`, replacing the `@request.auth.*` API rules seen throughout `apps/pocketbase/pb_migrations/*`.
- **Business logic** (assignment emails, default role on create, priority-order auto-numbering) moves into the **Server Actions themselves** (TypeScript), not Postgres triggers — e.g. `updateProperty()` checks if `employee_id` changed and sends the email in the same function, after the Supabase update call.
- **TanStack Query + Server Actions**: `queryFn`/`mutationFn` calls the action function directly (imported from `src/actions/*.ts`), not `fetch()`.
- **Strict layering**: all Supabase/Postgres code (`supabase.from(...)`, `.storage...`, joins, RLS-aware queries) lives *only* inside `src/actions/*.ts`. Pages and components never import a Supabase client directly — they call action functions (usually via a TanStack Query hook in `src/hooks/queries/`). This applies to every module below, not just the ones already built.
- **Realtime**: PocketBase's only realtime consumer (`ClientPipelineWidget.tsx`, wildcard-subscribing to `client_statuses`/`client_status_history`/`clients` purely to trigger a debounced refetch) maps to Supabase Realtime `postgres_changes` via one small shared hook, `useRealtimeInvalidate`, built once and reused wherever else realtime is needed.
- **IDs**: move from PocketBase's 15-char autogenerated text IDs to Postgres `uuid` (`gen_random_uuid()`) — every FK and every TS type is designed around `uuid` from the start rather than retrofitted.
- **File uploads**: `properties.images` (PocketBase file field, 12 files max, 20MB, jpeg/png/gif/webp) moves to a Supabase Storage bucket `property-images`, action-side validated with the same constraints, storing public URLs on the row.
- **Full cutover**: `apps/pocketbase` is deleted only once every module below has been migrated and verified end-to-end — not per-module.

## Foundational Work (must land before any module below)

### 1. Supabase project + client setup
- New Supabase project; env vars `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` in `apps/web/.env.local`.
- Add `@supabase/supabase-js`, `@supabase/ssr`, `@tanstack/react-query` to `apps/web/package.json`.
- `apps/web/src/lib/supabase/client.ts` — browser client (`createBrowserClient`), replaces `apps/web/src/lib/pocketbaseClient.ts`.
- `apps/web/src/lib/supabase/server.ts` — server client (`createServerClient`, cookie-based via `next/headers`), replaces `apps/web/src/lib/pocketbaseServer.ts`. Every Server Action imports this.
- `apps/web/src/middleware.ts` — rewritten with `@supabase/ssr`'s middleware session-refresh helper, keeping the existing master/company route-prefix protection logic but checking `profiles.role` (fetched via the refreshed session) instead of `authStore.model.role`/`.collectionName`.
- `apps/web/src/app/providers.tsx` (or extend the existing `Providers.tsx`) wraps the tree in `QueryClientProvider`.

### 2. Schema — all 19 tables in one SQL migration set
Field shapes below are taken directly from `apps/web/src/types/pocketbase.types.ts` (already hand-derived from the PocketBase migrations) and `apps/pocketbase/pb_migrations/*`, translated to Postgres:

| Table | Notes |
|---|---|
| `profiles` | **new** — bridges `auth.users.id` (PK/FK) to `role`, `company_id`, `employee_id`, `name`. Replaces `master_admins`, `company_super_admins`, `company_employees` as auth-bearing tables. |
| `companies` | `company_code`, `company_name`, `email`, `subscription_start_date`, `subscription_end_date`, `max_employee_count`, `is_active`, `is_frozen` |
| `employees` | `first_name`, `last_name`, `email`, `company_id`, `disabled`, `phone`, `job_title` (enum: وكيل مبيعات / مسؤول / مدير) — the base employee record `profiles.employee_id` points to |
| `property_types` | `name`, `company_id` |
| `client_statuses` | `name`, `company_id`, `priority_order` |
| `owner_statuses` | `name`, `company_id` |
| `owners` | `name`, `phone`, `country`, `company_id`, `marketing_channel`, `assigned_employee_id` |
| `properties` | `code`, `type` (FK → property_types), `land_area`, `building_area`, `emirate`, `area`, `owner_id` (FK), `price`, `commission_percentage`, `employee_id` (FK → profiles), `title`, `description`, `images` (text[] of Storage URLs), `listing_type`, `company_id`, `status`, `advertising_permit_number` |
| `clients` | `name`, `phone`, `country_code`, `interest_type`, `interested_properties` (uuid[] FK → properties), `employee_id`, `company_id`, `marketing_channel`, `follow_up_date`, `follow_up_time` |
| `client_status_history` | `client_id`, `status_id`, `note`, `created_by`, `created_by_name`, `company_id`, `transferred_from_employee`, `transferred_to_employee`, `employee_id`, `status`, `follow_up_date` |
| `owner_status_history` | `owner_id`, `status_id`, `note`, `created_by`, `created_by_name`, `company_id` |
| `property_status_history` | `property_id`, `status`, `note`, `created_by`, `created_by_name`, `company_id` |
| `marketing_channels` | `name`, `company_id` (the fixed `MARKETING_CHANNELS` const list stays a TS-level enum/check constraint) |
| `areas_districts` | `name`, `emirate`, `description`, `company_id` |
| `revenues` | `property_code`, `emirate`, `area_district`, `commission_value`, `employee_id`, `employee_name`, `deal_completion_date`, `client_id`, `client_name`, `owner_name`, `company_id` |
| `legal_pages` | `page_type` (privacy_policy / terms_of_service), `title`, `content`, `updated_by` |

Conventions applied to every table: `id uuid primary key default gen_random_uuid()`, FKs as `uuid references ...`, `created_at timestamptz default now()`, `updated_at timestamptz default now()` + trigger to keep it fresh. RLS enabled everywhere, policy pattern translated from the PocketBase API rules (e.g. `apps/pocketbase/pb_migrations/1780900194_002_created_properties.js`):
- `select`: `exists (select 1 from profiles where profiles.id = auth.uid() and profiles.company_id = <table>.company_id)`
- `insert`/`update`/`delete`: same, plus `profiles.role = 'company_super_admin'` (looser `select`-only access for `company_employee`, matching today's `listRule`/`viewRule` vs `createRule`/`updateRule`/`deleteRule` split)
- `master_admin` role: separate bypass policy, not company-scoped (needed for `companies` CRUD from the master dashboard).

### 3. Storage
- Bucket `property-images` (public read), replacing the `properties.images` PocketBase file field. Same constraints (12 files, 20MB, jpeg/png/gif/webp) enforced in the `createProperty`/`updateProperty` actions before upload.

### 4. Realtime
- `apps/web/src/hooks/useRealtimeInvalidate.ts` (new, shared) — `useRealtimeInvalidate(table, queryKey, filter?)` opens a `supabase.channel(...)` with `.on('postgres_changes', { event: '*', schema: 'public', table, filter }, debounced(() => queryClient.invalidateQueries({ queryKey })))`, cleaning up via `supabase.removeChannel` on unmount. Requires `alter publication supabase_realtime add table clients, client_statuses, client_status_history;` in the schema migration (the only tables that need it, matching today's only realtime consumer).

### 5. Business-logic hook replacements (from `apps/pocketbase/pb_hooks/*.pb.js`)
All of these move into the relevant Server Action, per the user's decision — not Postgres triggers:
- `property-assignment-email.pb.js`, `owner-assignment-email.pb.js`, `client-assignment-email.pb.js` → each becomes a check-and-send block inside `updateProperty`/`updateOwner`/`updateClient` (diff `employee_id`/`assigned_employee_id` against the previous row, send the same bilingual HTML email template). Needs an actual mail-sending mechanism decided (Resend/Postmark/SMTP from the Next.js server, or keep hitting the existing "Builder Mailer" HTTP API from `builder-mailer.pb.js` via its env vars `BUILDER_MAILER_SENDER_ADDRESS`/`BUILDER_MAILER_API_URL`/`BUILDER_MAILER_API_KEY` since that infra already exists independent of PocketBase) — **flagged as an open question, see below**.
- `set_company_employee_role.pb.js` (default `role` to `company_employee` on create) → default value in the `profiles` insert path of whatever action creates a company employee (`createEmployee` action), or a Postgres column default `role text default 'company_employee'` (this one is a pure data-default, not conditional business logic, so a column default is simplest and consistent with "infra stays in the DB, business logic stays in TS").
- `migrate-priority-order.pb.js`, `populate_owner_statuses_priority.pb.js` (auto-number `priority_order` on `client_statuses`/`owner_statuses` create if unset) → `createClientStatus`/`createOwnerStatus` actions compute the next `priority_order` (`count(*) + 1` scoped to `company_id`) before insert.
- `auto_populate_priority_orders.pb.js` → confirmed no-op stub in PocketBase already; no replacement needed.
- `external-dashboard.pb.js`, `custom-migrations-cmd.pb.js` → PocketBase-admin-UI/tooling only, no app-code equivalent needed; simply not carried over.

## Module-by-module migration (execution order)

Each module follows the same four steps: **(a)** Server Actions file in `apps/web/src/actions/`, **(b)** TanStack Query hooks file in `apps/web/src/hooks/queries/`, **(c)** rewire the consuming pages/components (swap `pb.collection(...)` calls for the query hooks; most forms are *already* on react-hook-form + zod from the earlier migration, so only the data-fetching/mutation half changes), **(d)** verify (`tsc --noEmit`, `npm run build`, manual smoke test of that module's CRUD + any RLS boundary).

1. **Auth & Profiles** — rewrite `CompanyAuthContext.tsx` and `MasterAuthContext.tsx` to use `supabase.auth.signInWithPassword` + a `profiles` fetch for role/company_id and the existing frozen/inactive/subscription-expired business checks (same error messages). Required before any other module's actions can be exercised through the UI.
2. **Properties** — `actions/properties.ts`, `hooks/queries/useProperties.ts`; rewires `app/properties/page.tsx`, `components/PropertyCard.tsx`, `components/PropertyDetailModal.tsx`. Covers Storage upload and one assignment-email action.
3. **Owners** — `actions/owners.ts`, `hooks/queries/useOwners.ts`; rewires `app/owners/page.tsx`, `components/OwnerDetailModal.tsx`, `components/OwnerStatusesTab.tsx`, `hooks/useOwnerStatusBadge.ts`. Covers `owner_status_history` and the owner assignment email.
4. **Clients** — `actions/clients.ts`, `hooks/queries/useClients.ts`; rewires `app/clients/page.tsx`, `components/ClientDetailModal.tsx`, `components/ClientPipelineWidget.tsx` (first real consumer of `useRealtimeInvalidate`), `components/ClientsBySourceWidget.tsx`, `components/FollowUpCalendarWidget.tsx`, `hooks/useStatusUpdate.ts`. Covers `client_status_history` and the client assignment email.
5. **Employees** — `actions/employees.ts`, `hooks/queries/useEmployees.ts`; rewires `app/employees/page.tsx`, `app/company-dashboard/employees/page.tsx` + `/new` + `/[id]`, `components/EmployeeModal.tsx`, `components/EmployeeReassignmentModal.tsx`, `components/EmployeeDeletionDialog.tsx`, `components/EmployeeLeaderboard.tsx`, `hooks/useEmployeeCount.ts`, `hooks/useEmployeeDeletion.ts`. Covers the default-role-on-create logic and the `employees`/`profiles` split.
6. **Companies** — `actions/companies.ts`, `hooks/queries/useCompanies.ts`; rewires `app/master-dashboard/page.tsx`, `app/master-dashboard/companies/page.tsx` + `/new` + `/[id]`, `components/SubscriptionRenewalForm.tsx`. Master-realm RLS bypass exercised here.
7. **Settings entities** (`property_types`, `client_statuses`, `owner_statuses`, `marketing_channels`, `areas_districts`) — `actions/settings.ts`, `hooks/queries/useSettings.ts`; rewires `app/settings/page.tsx` (5 CRUD tabs), `components/MarketingChannelsTab.tsx`, `components/AreasDistrictsTab.tsx`, `components/OwnerStatusesTab.tsx`. Covers the priority-order auto-numbering actions.
8. **Revenue** — `actions/revenues.ts`, `hooks/queries/useRevenues.ts`; rewires `app/revenue/page.tsx`, `components/DealCompletedModal.tsx` (writes `properties.status`, `revenues`, and `property_status_history` in one action, `completeDeal`).
9. **Legal Pages** — `actions/legalPages.ts`, `hooks/queries/useLegalPages.ts`; rewires `app/admin/legal-pages/page.tsx`, `app/privacy-policy/page.tsx`, `app/terms-of-service/page.tsx`.

Each action returns a typed `{ data } | { error }` result (never throws across the server/client boundary), so calling hooks surface `t()`-translated errors consistent with how the zod-schema forms already work.

## Decommissioning (only after step 9 above is verified)
- Delete `apps/pocketbase/` entirely (binaries, `pb_data`, `pb_hooks`, `pb_migrations`).
- Root `package.json`: remove `apps/pocketbase` from the `dev`/`start` `concurrently` invocations (become plain `next dev`/`next start` passthroughs); drop it from the `workspaces` glob if it stops matching anything meaningful.
- `apps/web/package.json`: remove the `pocketbase` npm dependency.
- Delete `apps/web/src/lib/pocketbaseClient.ts`, `pocketbaseServer.ts`.
- Delete `apps/web/src/types/pocketbase.types.ts`, replaced by Supabase-generated types (`supabase gen types typescript --project-id ... > src/types/supabase.types.ts`) — every action/hook file written above should already import from this new file, so this is a rename/cleanup step, not a rewrite.

## Open question before execution starts

**Mail delivery**: the 3 assignment-email hooks currently go through PocketBase's `$app.newMailClient().send()`, which (per `builder-mailer.pb.js`) gets intercepted and routed to a "Builder Mailer" HTTP API when SMTP isn't configured. Once PocketBase is gone, something else must send these emails from the Server Actions. Two direct options: (a) call the same Builder Mailer HTTP API directly from the action (reuses existing infra/env vars, fastest), or (b) adopt a standard transactional-email provider (Resend, Postmark, etc.) wired into Supabase or called directly from Next.js. This should be confirmed before the Properties/Owners/Clients modules are executed, since all three need it.
