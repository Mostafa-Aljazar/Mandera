# Mandera CRM — Project Overview & Architecture

## What this project is

**Mandera CRM** is a **multi-tenant, bilingual (Arabic/English) SaaS CRM built specifically for real-estate brokerages in the UAE**. It is not a generic CRM — its data model and workflows are purpose-built around the real-estate brokerage business:

- **Properties**: listings with price, commission %, land/building area, `emirate`/`area` (UAE-specific location fields), a UAE RERA/DLD-style **advertising permit number**, listing type (sale/rent), and up to 12 images.
- **Owners**: landlords/sellers who own the properties, tracked with their own status pipeline (e.g. new lead → negotiating → listed) and marketing-channel attribution.
- **Clients**: buyers/tenants (leads), each interested in up to 4 properties, tracked through a sales pipeline with a full status-change history and follow-up scheduling.
- **Employees**: sales agents/officers/managers, each assigned to specific owners/properties/clients — the CRM enforces that regular employees only see records assigned to them, while a "company super admin" sees everything for their company.
- **Revenue**: a commission/deal ledger recording completed transactions, visible only to company admins (regular agents can't see commission figures).
- **Marketing attribution**: every owner and client records which channel brought them in (Google, Facebook, Instagram, TikTok, Property Finder, Bayut, Dubizzle, Website, etc. — the major channels used in UAE real-estate marketing).

### Who uses it (two-tier SaaS structure)
1. **Master Admin (platform level)** — Mandera itself, the SaaS operator. Master admins create and manage tenant companies (brokerages), set their subscription dates, employee limits, and can freeze/unfreeze a tenant's access. They also manage the platform's public legal pages (Privacy Policy, Terms of Service).
2. **Company Super Admin (tenant level)** — the owner/manager of a subscribing real-estate brokerage. Manages their own employees, properties, owners, clients, and views revenue/commission reports.
3. **Company Employee (tenant level)** — a sales agent working under a brokerage. Sees and manages only the owners/properties/clients assigned to them.

This is a classic **B2B SaaS** shape: one product, sold to multiple independent real-estate companies (tenants), each with isolated data, its own team, and its own subscription — all enforced at the database level via row-level multi-tenancy rules keyed on `company_id`.

### Scope in one line
A subscription-based back-office platform that lets real-estate brokerages in the UAE manage their property listings, owner/client relationships, sales-agent assignments, and commission tracking, in Arabic and English, while letting the platform operator (Mandera) onboard and bill multiple brokerage tenants from a single admin dashboard.

# Architectural Deep-Dive: `apps/web` (Mandera CRM Frontend)

## 0. Critical framing correction

Despite being asked about "Next.js frontend," **`apps/web` is not a Next.js application**. There is no `app/` or `pages/` router directory, no `next.config.js`, and `next` does not appear anywhere in `package.json`. It is a **Vite-powered React Single Page Application (SPA)** using **React Router v7** for client-side routing.

Strong evidence this project was scaffolded/generated by **Hostinger Horizons** (an AI app-builder product):
- `index.html` → `<meta name="generator" content="Hostinger Horizons" />`
- `public/.htaccess` → `Header set X-Powered-By "Hostinger Horizons"`
- `apps/web/plugins/` — an entire custom Vite plugin suite (`visual-editor`, `selection-mode`, `site-pages`, `vite-plugin-pocketbase-auth.js`) that implements an in-browser visual/inline editor, dev-time error reporting to a parent iframe (`window.parent.postMessage`), and auth-token bridging from a Horizons preview host (`horizons.hostinger.com` / `horizons.hostinger.dev`). This is builder/tooling infrastructure, not product code.
- `vite.config.js` injects runtime error/console interceptors that `postMessage` to `window.parent` — classic embedded-iframe-preview tooling.

This matters for the "educational explanation": the app's *product* code lives almost entirely in `apps/web/src/`; everything in `apps/web/plugins/` and the Horizons-specific bits of `vite.config.js` is generator/editor scaffolding, safely ignorable for understanding business logic.

### 2a. Where routes live (no filesystem routing — explicit route table)

Since there's no App/Pages Router, routing is 100% defined in **`src/App.jsx`** via `<Routes>`/`<Route>` from `react-router-dom`. Every page component lives flatly in `src/pages/*.jsx` and is manually wired to a path string. Route groups by audience:

- **Public:** `/` (Home), `/privacy-policy`, `/terms-of-service`
- **Master Admin** (platform operator, manages tenant companies): `/master-login`, `/master-dashboard`, `/master-dashboard/companies`, `/master-dashboard/companies/new`, `/master-dashboard/companies/:id`, `/admin/legal-pages` — all wrapped in `<ProtectedMasterRoute>`
- **Company Admin/Employee** (tenant CRM users): `/company-login`, `/company-dashboard`, `/company-dashboard/employees(/new|/:id)`, `/employees`, `/settings`, `/owners`, `/properties`, `/properties/:id`, `/clients`, `/revenue` — all wrapped in `<ProtectedCompanyRoute>`
- Catch-all `*` → `<Navigate to="/" replace />`

This is a **multi-tenant SaaS CRM** with two separate auth realms: a platform-level "master admin" that manages company tenants/subscriptions, and per-company admins/employees who use the actual CRM (clients, properties, owners, revenue).

## 3. Frontend ↔ backend communication (PocketBase)

**Client initialization — `apps/web/src/lib/pocketbaseClient.js`:**
```js
import Pocketbase from 'pocketbase';
const POCKETBASE_API_URL = '/hcgi/platform';
const pocketbaseClient = new Pocketbase(POCKETBASE_API_URL);
export default pocketbaseClient;
export { pocketbaseClient };
```
This is a **singleton PocketBase client**, imported everywhere as `import pb from '@/lib/pocketbaseClient'`. Notably the base URL is the **relative path `/hcgi/platform`**, not an absolute URL and not an env var — meaning in production the web app and PocketBase are expected to sit behind the same origin/reverse proxy, with `/hcgi/platform` proxied through to the PocketBase server (this is a Hostinger Horizons hosting convention). There is no `.env`/`VITE_POCKETBASE_URL` anywhere in application code.

**No app/api routes exist** — impossible by construction since this is a Vite SPA, not Next.js. All "API calls" are just direct PocketBase SDK calls made from React components/pages, e.g. in `ClientsPage.jsx`:
```js
pb.collection('clients').getFullList({ filter: finalFilter, expand: 'employee_id,interested_properties', sort: '-created', $autoCancel: false })
pb.collection('clients').create(payload, { expand: '...' })
pb.collection('clients').update(activeClient.id, payload, { ... })
```
Data-fetching pattern throughout the codebase (seen identically in `ClientsPage.jsx`, `PropertiesPage.jsx`, etc.):
1. A `fetchData()` async function inside the page component builds a PocketBase `filter` string by hand-concatenating conditions (e.g. `company_id = "..." && employee_id = "..."`) — **manual multi-tenant scoping is enforced client-side per query**, not via a shared data layer.
2. Called from a `useEffect` keyed on dependencies like `[company?.id, currentUser?.id, filterState]`.
3. Results stored in local `useState` (no cache/store).
4. Files/images are resolved via `pb.files.getUrl(record, filename, { thumb: '400x300' })` (see `PropertyCard.jsx`).
5. `$autoCancel: false` is passed on nearly every call to prevent PocketBase's SDK from auto-cancelling duplicate in-flight requests (a recurring team convention, applied very consistently — good signal of copy-paste-driven consistency across the codebase).

**The actual backend** lives in the sibling app `apps/pocketbase/` — a self-hosted PocketBase binary (`pocketbase` executable, `.pocketbase-version` = `0.38.0`) with:
- `pb_migrations/` — ~90 JS migration files defining collections: `companies`, `master_admins`, `company_super_admins`, `employees`, `company_employees`, `owners`, `properties`, `clients`, `client_statuses`/`client_status_history`, `owner_statuses`/`owner_status_history`, `property_status_history`, `property_types`, `marketing_channels`, `revenues`, `legal_pages`, `app_settings`.
- `pb_hooks/` — server-side JS hooks (`*.pb.js`) for things like assignment emails (`client-assignment-email.pb.js`, `owner-assignment-email.pb.js`, `property-assignment-email.pb.js`), auto-populating priority order fields, role-setting, and an `external-dashboard.pb.js` custom endpoint.
- `pb_data/` — the live SQLite data files (`data.db`) and generated `types.d.ts`/`database-types.d.ts` (TypeScript ambient declarations *for writing PocketBase hook/migration scripts in an editor*, unrelated to the web app's own typing).

So architecturally: **Vite React SPA → PocketBase JS SDK → self-hosted PocketBase (Go binary + SQLite) → JS migrations/hooks for schema and business-rule automation (emails, etc.)**. There is no separate custom REST/GraphQL API layer — PocketBase's auto-generated REST API *is* the backend API, secured by PocketBase's declarative collection rules (defined in the migration files, e.g. `1781069462_003_update_rules_for_clients.js`).

## 4. Authentication

Two parallel, independent auth contexts, both backed by PocketBase's built-in auth collections and its client-side `authStore` (which persists to `localStorage` automatically via the SDK):

**`src/contexts/MasterAuthContext.jsx`** — for the `master_admins` PocketBase auth collection:
```js
const authData = await pb.collection('master_admins').authWithPassword(email, password, { $autoCancel: false });
if (authData.record.role !== 'master_admin') { pb.authStore.clear(); throw new Error('Invalid credentials'); }
```

**`src/contexts/CompanyAuthContext.jsx`** — more elaborate, since a "company login" could be either a `company_super_admin` or a `company_employee`. It tries the first collection, falls back to the second:
```js
try {
  authData = await pb.collection('company_super_admins').authWithPassword(email, password, { $autoCancel: false });
} catch (errAdmin) {
  authData = await pb.collection('company_employees').authWithPassword(email, password, { $autoCancel: false });
}
```
After authenticating it does extra business-rule validation client-side: fetches the related `companies` record and checks `is_frozen`, `isActive`, and whether `subscriptionEndDate` has passed — throwing user-facing errors (bilingual) and clearing the auth store if the tenant's subscription/account state is invalid. This is enforcing **SaaS billing/subscription gating in the frontend auth layer**, which is worth flagging to the owner as something that should also be enforced server-side (PocketBase API rules) since client-side checks alone aren't a security boundary.

**Route protection** is done via wrapper components, not Next.js middleware (there's no `middleware.ts` — can't exist in a Vite SPA). `src/components/ProtectedMasterRoute.jsx` and `src/components/ProtectedCompanyRoute.jsx` both follow the same pattern: read `isAuthenticated`/`initialLoading` from their respective context, show a spinner while loading, and `<Navigate to="/..." replace />` if unauthenticated:
```jsx
const ProtectedCompanyRoute = ({ children }) => {
  const { isAuthenticated, initialLoading } = useCompanyAuth();
  if (initialLoading) return <Spinner />;
  if (!isAuthenticated) return <Navigate to="/company-login" replace />;
  return children;
};
```
A third file, `ProtectedRoute.jsx`, also exists in `src/components/` (likely an older/generic variant — worth checking if still referenced).

Session bootstrapping on app load (in both contexts' `useEffect`) reads `pb.authStore.model` / `pb.authStore.isValid` synchronously (PocketBase SDK restores the token from `localStorage` on client construction), then re-validates role/collection membership before trusting it.

## 5. State management

There is **no Redux, Zustand, or TanStack Query/SWR** anywhere in the dependency tree or code. State management is deliberately minimal and idiomatic React:

- **React Context** for cross-cutting concerns: `LanguageContext` (i18n/RTL), `MasterAuthContext`, `CompanyAuthContext` — each a simple `createContext` + provider + custom hook (`useCompanyAuth()`, etc.) pattern, no reducer/middleware.
- **Local component state (`useState`/`useEffect`)** for all server data — every page owns its own `fetchData()` and refetches on demand (e.g., calling `fetchData()` again after a create/update mutation rather than using any cache invalidation library).
- No memoized/derived global store; filtering, search, and pagination state are page-local `useState` objects (e.g., `filterState` in `ClientsPage.jsx`).
- Custom hooks under `src/hooks/` encapsulate small reusable bits of derived/async state (e.g., `useSubscriptionStatus.js`, `useEmployeeCount.js`, `useStatusUpdate.js`, `useOwnerStatusBadge.js`, `use-toast.js` for the shadcn toast system, `use-mobile.jsx` for responsive breakpoint detection).

This means data-fetching, caching, and race-condition handling are all hand-rolled per page rather than centralized — a notable architectural characteristic (simpler to read, but no automatic caching/dedup/retries you'd get from React Query).

## 7. TypeScript usage

**There is effectively none in the frontend application.** Confirmed by:
- `jsconfig.json` (not `tsconfig.json`) at `apps/web/` root — this is the JS-only project-config equivalent, providing only the `@/*` → `src/*` path alias for editor IntelliSense, no type-checking config.
- `components.json` explicitly sets `"tsx": false`.

The `database-types.d.ts` and `pb_data/types.d.ts` files that *do* exist in the repo live in **`apps/pocketbase/`**, not `apps/web/`, and are TypeScript **ambient declaration files for writing PocketBase's own JS hooks/migrations** in an editor with better autocomplete (they declare globals like `$app`, `routerAdd`, `cronAdd` used inside `pb_hooks/*.pb.js`) — they're unrelated to the React frontend's type system.

There's no shared `types/` folder, no Zod schemas used for parsing (despite Zod being installed), and no JSDoc typedefs observed in the pages reviewed. This is worth flagging to the project owner as a potential improvement area (no compile-time safety on PocketBase record shapes).

## 10. Representative feature deep-dive: Clients, Properties, and forms

Reviewed `ClientsPage.jsx` + `ClientDetailModal.jsx` (Clients/Leads), `PropertiesPage.jsx` + `PropertyCard.jsx` (Properties), and `CompanyFormPage.jsx` (Company creation) as representative CRM entity implementations. The pattern is **extremely consistent** across all of them:

### Page-level structure (`src/pages/ClientsPage.jsx`, `PropertiesPage.jsx`)
1. Import a domain-specific header (`CompanyAdminHeader`), detail modal, and shared filter components (`FilterPanel`, `FilterChips`).
2. Pull `company`/`currentUser` from `useCompanyAuth()` — used for multi-tenant scoping and role-based query filtering.
3. Local `useState` for: the main list (`clients`/`properties`), loading flag, related lookup lists fetched in parallel (employees, statuses, marketing channels / property types, owners, areas), the currently "active" record for the detail modal, and a `filterState` object.
4. A single `fetchData()` async function that:
   - Guards on `company?.id && currentUser?.id`.
   - Fetches lookup collections in parallel via `Promise.all([...])`.
   - Builds a PocketBase filter string by array-pushing conditions then `.join(' && ')` — role-based enforcement is baked directly into this filter (`if (currentUser.role === 'company_employee') filterParts.push('employee_id = "..."')`), i.e., **employees only ever see their own assigned clients/properties, enforced by an additional filter clause added client-side** (mirrored — hopefully — by PocketBase API rules server-side).
   - Calls `pb.collection('clients').getFullList({ filter, expand, sort, $autoCancel: false })`.
   - Optionally does a **second query + client-side post-filter** for cases PocketBase's flat filter syntax can't express well (e.g. filtering by "current" status requires fetching all `client_status_history` records and manually reducing to the latest-per-client in JS, since status is a mutable history table rather than a column).
5. `useEffect(() => { fetchData(); }, [company?.id, currentUser?.id, filterState, t])` — refetch on any of those changing.
6. Render: `<Helmet>` for page `<title>`, header component, loading/empty states, then a responsive `grid` of `Card` components (shadcn) mapped from the data array, each opening a detail modal `onClick`.
7. CRUD mutations (`handleSaveClientInfo`) call `pb.collection(...).create/update(...)` directly inline in the page component (not abstracted into a service/repository layer), followed by a `toast.success/error` (sonner) and a full `fetchData()` refetch — no optimistic updates or cache patching.

### Forms and validation
This is the most notable architectural finding: **despite `react-hook-form`, `@hookform/resolvers`, and `zod` being installed, none of the reviewed feature pages/modals actually use them.** `ClientDetailModal.jsx` and `CompanyFormPage.jsx` both use plain **controlled `useState` objects** with manual `onChange` handlers:
```jsx
const [formData, setFormData] = useState({ name: client?.name || '', phone: client?.phone || '', ... });
<Input value={formData.name} onChange={e => setFormData({...formData, name: e.target.value})} />
```
Validation is manual and minimal, done right before submit and surfaced via toast:
```js
const handleSaveInfo = () => {
  if (!formData.name || !formData.phone || !formData.employee_id) {
    toast.error(t('Name, phone, and assigned agent are required.'));
    return;
  }
  onSaveInfo(finalData);
};
```
Server-side/PocketBase validation errors are caught and surfaced generically:
```js
if (err.response?.data) {
  const fieldErrors = Object.entries(err.response.data).map(([k, v]) => `${k}: ${v.message}`).join(', ');
  errorMsg += ` (${fieldErrors})`;
}
```
The only place `react-hook-form`/Zod machinery is *wired up* at all is inside the generic shadcn primitive `src/components/ui/form.jsx` (the boilerplate `Form`, `FormField`, `FormItem` context-provider components shadcn's CLI generates by default) — but a search across `src/` confirms **no page actually calls `useForm()` or `zodResolver()` to consume it**. This is a good, concrete thing to know: the scaffolding for robust schema-validated forms exists in the codebase (installed, generated) but the team's actual pages use a simpler, more manual "spread state + inline `if` checks" convention instead — consistent everywhere, but without centralized/reusable validation schemas.

### Component reuse pattern
Domain "cards" (`PropertyCard.jsx`) are small, presentation-focused, receive data + callback props (`onEdit`, `onView`, `onStatusChange`) and compute simple derived values inline (status-to-color maps as plain objects, e.g. `STATUS_COLORS`), pulling images via `pb.files.getUrl(record, filename, { thumb: '400x300' })` — PocketBase's built-in file/thumbnail serving, used directly in `<img src>`, no separate CDN/image-optimization layer (unlike Next.js's `<Image>` — again reinforcing this isn't Next.js).

Bilingual text throughout uses a `t(key, fallbackOrArabic)` call pattern from `react-i18next`, often passing the Arabic string directly as the second argument rather than always resolving through `locales/ar.json`/`en.json` keys — a mixed/pragmatic i18n convention worth noting (some strings are properly keyed in the locale JSON files, others are inlined bilingual literals directly in JSX).

# Deep Dive: `apps/pocketbase` — Mandera CRM Backend

## Overview

`apps/pocketbase` is the entire backend/database layer of this monorepo. Telling signs throughout the code (`horizons` CLI subcommands, `app-preview.com` URLs, `horizons-static-cdn.hostinger.com`) indicate this project was scaffolded/is managed by **Hostinger Horizons**, a platform that generates and hosts full-stack apps (a Vite/React frontend in `apps/web` + this PocketBase backend), deployed on Hostinger infrastructure.

No `.env` files exist anywhere — confirming environment variables (`PB_ENCRYPTION_KEY`, `PB_SUPERUSER_EMAIL`, `PB_SUPERUSER_PASSWORD`, `BUILDER_MAILER_*`) must be supplied externally (by the hosting platform/Hostinger Horizons at runtime), not committed to the repo. No `.gitignore` found either.

```json
  "scripts": {

    "start": "./pocketbase serve --http=0.0.0.0:8090 --encryptionEnv=PB_ENCRYPTION_KEY --dir=/data --migrationsDir=./pb_migrations --hooksDir=./pb_hooks --hooksWatch=false",
    "migrations:up": "./pocketbase horizons migrations:up --encryptionEnv=PB_ENCRYPTION_KEY --dir=./pb_data --migrationsDir=./pb_migrations --hooksDir=./pb_hooks --hooksWatch=false",
    "migrations:revert": "./pocketbase horizons migrations:revert --encryptionEnv=PB_ENCRYPTION_KEY --dir=./pb_data --migrationsDir=./pb_migrations --hooksDir=./pb_hooks --hooksWatch=false",
    "migrations:snapshot": "mkdir -p ./pb_snapshots && echo 'y' | ./pocketbase migrate collections --encryptionEnv=PB_ENCRYPTION_KEY --dir=./pb_data --migrationsDir=./pb_snapshots",
    "update": "./pocketbase update --encryptionEnv=PB_ENCRYPTION_KEY --backup=0"
  }
}
```

- **No npm dependencies at all** — no `node_modules` needed for this app. The "backend" is literally the `pocketbase` binary plus plain `.js` migration/hook files that PocketBase's embedded JSVM (goja) interprets directly — no build step, no bundler, no TypeScript compilation.
- `dev`: runs the binary directly against the default `pb_data` dir (relative), listening on all interfaces port 8090, with an encryption key read from env var `PB_ENCRYPTION_KEY`, and hooks file-watching disabled.
- `start` (production): same, but explicitly points `--dir=/data` (an absolute path — implies a mounted volume in production/Hostinger container), and explicit `--migrationsDir`/`--hooksDir`.
- `migrations:up` / `migrations:revert`: call a **custom CLI subcommand** named `horizons` (added via `pb_hooks/custom-migrations-cmd.pb.js` — see section 4) rather than PocketBase's built-in `migrate up/down`. This custom command fails loudly (exit code 1) if any migration fails — useful for CI/CD gating.
- `migrations:snapshot`: dumps the *current* live collections schema into a new migration file under `./pb_snapshots` using PocketBase's built-in `migrate collections` snapshot command (auto-confirmed via piped `'y'`).
- `update`: self-updates the PocketBase binary in place (`--backup=0` skips taking a backup first).

**Root monorepo `package.json`** (`mandera-crm/package.json`) orchestrates both apps via `concurrently` (plain npm workspaces, `workspaces: ["apps/*"]`, no Turborepo):
```json
"dev":   "concurrently --kill-others --raw \"npm run dev --prefix apps/web\" \"npm run dev --prefix apps/pocketbase\"",
"start": "concurrently --kill-others --raw \"npm run start --prefix apps/pocketbase\""
```
So `npm run dev` at the repo root starts the Vite/React frontend (`apps/web`) and PocketBase together in local development; `npm run start` at the root only boots PocketBase (implying the frontend is a static build served separately in production, or served via that `external-dashboard` hook — see below).

## 3. Database schema reconstruction (from `pb_migrations/`)

This is a **multi-tenant real-estate CRM** (Arabic/English bilingual, UAE-focused — fields like `emirate`, Arabic email bodies, AED-style commission fields). Tenancy is enforced by a `company_id`/`companyId` field on almost every collection, checked against `@request.auth.companyId` in API rules.

All base collections use a 15-character autogenerated text ID (`[a-z0-9]{15}`), plus standard `created`/`updated` autodate fields, unless noted.

### Auth collections (type: `auth`)

| Collection | Purpose | Key fields | Rules (final state) |
|---|---|---|---|
| **`_superusers`** | Built-in PB superuser table | — | authAlert disabled (see migrations) |
| **`master_admins`** | Platform-level super-admin (manages all companies) | `name` (text), `role` (text, e.g. `"master_admin"`) | list: `role='master_admin'`; view/update: self; create: nobody (`null`, i.e. must be seeded via migration) |
| **`company_super_admins`** | Per-company admin/owner account | `name`, `companyId` → relation→`companies`, `companyCode` (text), `role` (text) | list/view: self or same company; create/delete: `master_admin`; update: self |
| **`employees`** | HR-style employee master record (distinct from login accounts) | `firstName`, `lastName`, `email`, `companyId` → relation→`companies`, `disabled` (bool), later added: `phone` (required text), `job_title` (select: وكيل مبيعات / مسؤول / مدير i.e. Sales Agent / Officer / Manager) | list/view: same company; create/update/delete: `company_super_admin` of that company |
| **`company_employees`** | The actual **login/auth account** for an employee, linking `employees` ↔ `companies`; this is what agents authenticate as | `name`, `companyId` → relation→`companies`, `employeeId` → relation→`employees`, `role` (text, required — defaulted to `'company_employee'` via hook if not set) | list: same company; view: self or same company; create/delete: `company_super_admin`; update: self |

Note the odd duplication: **`employees`** (HR profile) is separate from **`company_employees`** (auth/login identity) but 1:1 linked via `employeeId`. Various downstream relations (e.g. `properties.employee_id`, `clients.employee_id`) actually point to `company_employees` (the login identity), while `owners.assigned_employee_id` points to `company_employees` too (via collection id `pbc_2029377502`, confirmed as `company_employees`).

### Core tenant/business collections (type: `base`)

**`companies`** (id `pbc_2313907151`) — the tenant table (SaaS-style)
- `companyCode` (text, unique index `idx_companies_companyCode`), `companyName`, `email`, `subscriptionStartDate`/`subscriptionEndDate` (date), `maxEmployeeCount` (number, min 1), `isActive` (bool), `is_frozen` (bool, added later — likely a "suspend tenant" flag)
- Rules: list/create/update/delete restricted to `master_admin`; view: `master_admin` OR the record's own `companyId` matches the requester's company (self-service view for `company_super_admin`s of their own tenant).

**`property_types`** (id `pbc_5678146546`) — lookup table per company (e.g. Villa, Apartment)
- `name`, `company_id` → relation→`companies`

**`client_statuses`** (id `pbc_1601181164`) — pipeline stages for clients, per company
- `name`, `company_id` → relation→`companies`, `priority_order` (number, min 1, added later — used for sort/kanban ordering, auto-populated by a hook)

**`owner_statuses`** (id `pbc_6814664971`) — pipeline stages for property owners, per company
- `name`, `company_id` → relation→`companies`
- **Interesting churn**: `priority_order` was added (`1781878081`) then **removed** (`1782779621`) — unlike `client_statuses` which kept it. Current schema: **no `priority_order`** on `owner_statuses`.

**`owners`** (id `pbc_6601306559`) — property owners (landlords/sellers)
- `name`, `phone`, `country`, `company_id` → relation→`companies`, `marketing_channel` (select: Google/Facebook/Instagram/TikTok/Snapchat/X/LinkedIn/Property Finder/Bayut/Dubizzle/Marjan/OpenSouk/Website — added later), `assigned_employee_id` → relation→`company_employees` (added later)
- Rules: create now open to any authenticated user (`@request.auth.id != ""` — loosened from originally `company_super_admin`-only); list/view/update/delete still company-scoped, update/delete restricted to `company_super_admin`.

**`properties`** (id `pbc_3656272165`) — real-estate listings
- `code`, `type` → relation→`property_types`, `land_area`/`building_area` (number), `emirate` (text), `area` (text), `owner_id` → relation→`owners`, `price` (number), `commission_percentage` (number), `employee_id` → relation→`company_employees`, `title`, `description`, `images` (file, up to 12, 20 MB max each, jpeg/png/gif/webp, with thumbs), `listing_type` (text — e.g. sale/rent), `company_id` → relation→`companies`; added later: `status` (text), `advertising_permit_number` (text — UAE RERA/DLD-style permit field)
- Rules (final): list restricted to the assigned employee (`employee_id = @request.auth.id`) or a `company_super_admin`; view: whole company; update: assigned employee or super admin; create/delete: super admin only.

**`clients`** (id `pbc_7825922583`) — leads/buyers/tenants
- `name`, `phone`, `country_code`, `interest_type` (text — buy/rent etc.), `interested_properties` → relation→`properties` (multi, max 4), `employee_id` → relation→`company_employees`, `company_id` → relation→`companies`; added later: `marketing_channel` (same select list as owners), `follow_up_date` (date), `follow_up_time` (text)
- Rules: create allows any authenticated user in the same company (`role = 'company_super_admin' || companyId = company_id`, effectively any company member since both sides check `company_id`); list/view/update/delete otherwise scoped to `company_super_admin` for mutation, whole company for read.

### History / audit-trail collections

**`client_status_history`** (id `pbc_8642255691`)
- `client_id` → relation→`clients`, `status_id` → relation→`client_statuses`, `note` (text), `created_by` → relation→`company_super_admins` *originally*, later **redefined as plain text** (see field-type churn below), `company_id` → relation→`companies`; added later: `employee_id` → relation→`company_employees`, `status` (text), `follow_up_date` (date), `transferred_from_employee` / `transferred_to_employee` → relation→`company_employees` (supports lead re-assignment/transfer tracking)
- **Field churn on `created_by`**: went from `relation→company_super_admins` (original) → removed → re-added as **text** (`1781070749`) → rules updated repeatedly. Final createRule: `companyId = company_id && (role='company_super_admin' || (auth.id = created_by && auth.collectionName = 'company_employees'))` — i.e. either a super admin, or the employee who owns this history record, can create it. updateRule/deleteRule: `null` originally (immutable audit log) in the original migration, but later opened to `role='company_super_admin' || auth.id = created_by`.
- Only a `listRule`/`viewRule` (company-scoped) — `updateRule: null` in the very first migration reflects the intent that this is meant to be an append-only audit trail, though later migrations do add update/delete rules (some drift from that original intent).

**`owner_status_history`** (id `pbc_1586259687`) — same shape/pattern as client_status_history but for owners
- `owner_id` → relation→`owners`, `status_id` → relation→`owner_statuses`, `note` (text, max 300), `created_by` (relation→`company_super_admins` → later churned to text, similar to above), `company_id` → relation→`companies`
- Final createRule: `companyId = company_id && (role='company_super_admin' || auth.id = owner_id.assigned_employee_id)` — i.e. super admin or the employee assigned to that owner.

**`property_status_history`** (id `pbc_6012716887`)
- `property_id` → relation→`properties`, `status` (text), `note` (text, max 300), `created_by` (same churn pattern: relation→`company_super_admins` → text), `company_id` → relation→`companies`
- Final rules: create/update/delete restricted to `company_super_admin` only (tighter than the other two history tables — earlier draft allowed the assigned employee too, but this was walked back).

### Financial

**`revenues`** (id `pbc_2482480816`) — completed-deal / commission ledger
- `property_code`, `emirate`, `area_district`, `commission_value` (number), `employee_id` → relation→`company_employees` (required initially, made optional later), `employee_name` (denormalized text snapshot), `deal_completion_date` (date), `client_id` → relation→`clients`, `client_name` (denormalized text), `owner_name` (denormalized text), `company_id` → relation→`companies`
- Notably denormalizes employee/client/owner names as plain text (likely so historical revenue records remain readable even if the referenced employee/client is later deleted/renamed).
- Rules: list/view restricted to `company_super_admin` only (regular employees cannot see revenue/commission data); `updateRule: null` (immutable once created).

### Misc / platform collections

**`marketing_channels`** (id `pbc_2844368336`) — a *custom* per-company marketing-channel lookup table (separate from the hardcoded `select` field used on `clients`/`owners` — possibly a planned migration away from hardcoded enum values toward a dynamic table)
- `name`, `company_id` → relation→`companies`

**`legal_pages`** (id `pbc_7509955575`) — CMS-style content for Privacy Policy / Terms of Service, globally editable by `master_admin`
- `page_type` (select: `privacy_policy` | `terms_of_service`), `title`, `content` (plain text, holds raw HTML), `updated_by` → relation→`users` (PocketBase's default `users` collection — note this predates/coexists with the custom auth collections)
- Rules: `listRule`/`viewRule` = `""` (public, world-readable — makes sense for legal pages shown to any visitor); update restricted to `master_admin`; create/delete unset (default = only superusers via API).
- Seeded with 2 default records (Privacy Policy, Terms of Service) via `1781234973_003_seed_legal_pages_2_records.js`.

## 4. `pb_hooks/` — custom server-side JS logic

All hooks use the `.pb.js` naming convention PocketBase auto-loads. Ten files total:

1. **`set_company_employee_role.pb.js`** — `onRecordCreate` for `company_employees`: defaults `role` to `"company_employee"` if not explicitly set on creation.

2. **`property-assignment-email.pb.js`** — `onRecordAfterUpdateSuccess` on `properties`: when `employee_id` changes to a new value, looks up the employee and sends a bilingual (Arabic/English) HTML email via `$app.newMailClient()` notifying them of the new property assignment. Sender: `crm@mandera.site`. Wrapped in try/catch so email failures never block the record update.

3. **`owner-assignment-email.pb.js`** — same pattern but for `owners.assigned_employee_id` changes, notifying the assigned employee about the new owner. Sender: `crm@mandera.site` / "Mandera CRM".

4. **`client-assignment-email.pb.js`** — same pattern for `clients.employee_id` changes, sending a bilingual client-assignment notification email.

5. **`external-dashboard.pb.js`** — `routerAdd("GET", "/{$}", ...)`: intercepts the root route and **proxies PocketBase's Admin UI from an external CDN** (`https://horizons-static-cdn.hostinger.com/pocketbase/v{VERSION}/ui/dist/index.html`, version read from `.pocketbase-version`) rather than serving the bundled local dashboard. This is why there's no `pb_public` folder — the admin UI itself is fetched live from Hostinger's CDN on every root request. Confirms this app is managed/hosted through **Hostinger Horizons**.

6. **`custom-migrations-cmd.pb.js`** — registers a new root CLI command group `horizons` with two subcommands used by `package.json`'s `migrations:up`/`migrations:revert` scripts:
   - `migrations:up` — shells out to invoke `./pocketbase migrate up` internally and **throws/exits non-zero** if any migration fails (the built-in command doesn't do this, so this is a CI/CD-friendliness patch).
   - `migrations:revert <files...>` — a bespoke "revert specific migrations by filename" tool (PocketBase's built-in revert only supports reverting the last N migrations by count, not by name). It parses each migration file's source, dynamically re-executes it with a stubbed `migrate()` capturing only the `down` function, runs all reverts inside a single DB transaction (`$app.runInTransaction`), deletes the corresponding rows from the internal `_migrations` tracking table, and then **deletes the migration files from disk** on success.

7. **`auto_populate_priority_orders.pb.js`** — `onRecordAfterCreateSuccess` hooks for `owner_statuses` and `areas_districts` (note: `areas_districts` collection doesn't appear to exist in any migration — likely dead/future code, or a collection created ad hoc outside migrations). Back-fills a `priority_order` on any records missing it, sequentially, ordered by creation date. (Somewhat contradicts the later migration `1782779621` that *removed* `priority_order` from `owner_statuses` — this hook may now be partially orphaned/no-op logic for that collection.)

8. **`builder-mailer.pb.js`** — `onMailerSend` hook: if PocketBase's built-in SMTP is **not** configured/enabled (`settings.smtp.enabled` false), intercepts all outgoing mail and instead POSTs it to an external "Builder Mailer" API (`${BUILDER_MAILER_API_URL}/api/v2/email`, authenticated via `BUILDER_MAILER_API_KEY` bearer token, `from` address from `BUILDER_MAILER_SENDER_ADDRESS`). **Key authorization design patterns observed:**
- **Multi-tenancy isolation**: nearly every rule starts with `@request.auth.companyId = company_id` (or `companyId`), meaning the authenticated user's own `companyId` claim must match the record's tenant — a classic row-level multi-tenant guard implemented entirely through PocketBase's rule DSL rather than application code.
- **Role-based elevation**: `company_super_admin` role generally required for destructive/administrative writes (create/update/delete on master data); regular `company_employees` are largely read-scoped, plus can update records assigned specifically to them (e.g. `properties.employee_id = @request.auth.id`).
- **Relation-traversal rules**: rules like `auth.id = owner_id.assigned_employee_id` or `property_id.employee_id` dot into related records directly inside the rule expression — a distinctive PocketBase capability (filter expressions can traverse relations).
- **Audit-log immutability intent, imperfectly enforced**: `client_status_history`, `owner_status_history`, `property_status_history`, and `revenues` were all *originally* designed with `updateRule: null` (fully immutable once written — a proper audit trail), but several were subsequently loosened via later migrations to allow the creator or a super admin to update/delete them, undermining that original audit-log guarantee somewhat.
- **Rule/relation-type churn**: the `created_by` field on all three `*_status_history` collections was originally a `relation` field pointing at `company_super_admins`, then removed, then re-added as a **plain `text`** field — meaning `created_by` today just stores a raw string (likely a record ID or name) with no referential integrity enforced by PocketBase, only convention.

## 10. Summary — backend

`apps/pocketbase` is the entire backend/database layer: a self-hosted, vanilla **PocketBase v0.38.0** binary (no custom Go code) whose schema is defined by **~84 JS migration files** and extended by **10 custom JS hooks** for email notifications, a managed-mailer proxy, an admin-UI CDN proxy, and custom migration CLI tooling — all clearly provisioned by the **Hostinger Horizons** platform. The database is **SQLite**, managed entirely through PocketBase's declarative collection/rule system rather than hand-written SQL or an ORM. Known rough edges: hardcoded demo credentials in seed migrations, some schema/rule churn (fields added then removed, immutability guarantees loosened over time), a couple of orphaned/dead hooks, and no documentation anywhere in the repo.
